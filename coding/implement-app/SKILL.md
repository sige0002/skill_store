# アプリ実装スキル

以下のリクエストに基づいてアプリケーションを実装する：

**Request:** $ARGUMENTS

## ワークフロー

### Phase 1: 要件分析
- リクエストを解析し、機能要件と非機能要件を抽出する
- 入出力、データモデル、外部依存を特定する
- 曖昧な点を列挙し、妥当なデフォルトで解決する（前提は明示する）

### Phase 2: アーキテクチャ設計
コードを書く前に **think hard** でアーキテクチャを設計する。

- 適切なパターン（MVC、レイヤード、イベント駆動など）を選ぶ
- モジュール境界とインターフェースを定義する
- ディレクトリ構成を計画する
- 依存を選定する — よくメンテされ、最小限のパッケージを優先
- 主要な設計判断はコードコメントとして残す

### Phase 3: 環境セットアップ
- Python プロジェクトでは `uv` を使用する — 素の `pip install` は絶対に使わない
  ```bash
  uv venv .venv && source .venv/bin/activate
  uv pip install <packages>
  ```
- Node.js プロジェクトでは `npm` または `pnpm` を使う
- 再現性のために依存バージョンをピン留めする

### Phase 4: 実装
- 小さく、テスト可能な単位で段階的に実装する
- 各モジュールには明確な docstring と型ヒント（Python）または JSDoc（JS/TS）を付ける
- 複雑なビジネスロジックや難しいアルゴリズムには **think hard** を使う
- 言語固有の慣用的な規約に従う（PEP 8、ESLint デフォルト等）
- エラーは明示的に処理する — 素の `except:` や握りつぶしは禁止

### Phase 5: テスト
- 実装と並行してユニットテストを書く
- カバー範囲：正常系、エッジケース、エラー条件
- Python は `pytest`、JS/TS は `vitest` または `jest`
- 全テストを実行し、次に進む前に失敗を修正する：
  ```bash
  uv run pytest -v   # Python
  npm test            # Node.js
  ```

### Phase 6: 検証
- サンプル入力で end-to-end 実行する
- 出力が期待通りか検証する
- lint/型エラーをチェックする：
  ```bash
  uv run ruff check . # Python
  npx tsc --noEmit    # TypeScript
  ```
- 作ったもの、実行方法、既知の制限をまとめる

## ガイドライン
- 新規作成より既存ファイル編集を優先する
- シークレット、認証情報、`.env` ファイルは絶対にコミットしない
- 関数は短く単一目的に保つ
- タスクが大きいときはサブタスクに分割し、反復的に実装する
