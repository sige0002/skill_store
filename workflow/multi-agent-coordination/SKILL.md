# マルチエージェント協調

## 概要

複数エージェントを並列で稼働させ、大規模タスクを効率的に遂行するための調整手法。フェーズ分割と役割分担により、エージェント間の衝突を防ぎつつ並列性を最大化する。

## 1. タスクの分析と分解

タスクを独立したサブタスクに分解する：

- 全体のゴールと最終成果物を特定する
- ゴール達成に必要な作業項目を列挙し、依存関係を判断する
- 並列実行可能な独立項目をグループ化し、逐次実行が必要な項目を特定する

**重要な問い：**
- 状態共有なしに独立して実施可能なサブタスクはあるか？
- 他の作業をブロックするクリティカルパスはどれか？
- 競合を引き起こす可能性のある共有リソース（ファイル・設定）はあるか？

## 2. フェーズ設計

1. **依存関係でフェーズを区切る** — 後続が前段の成果物を必要とする場合は別フェーズにする。並列化の粒度を依存関係で決めることで手戻りを防ぐ
2. **独立タスクは並列** — 1フェーズ内で依存のないタスクは全て並列投入する
3. **統合検証はフェーズ間に挟む** — 監督役が全テスト + 型チェックを実行し、早期にバグを発見して後段への波及を防ぐ

典型的な3フェーズ構成：

```
Phase 1: 基盤整備（テスト・設定・型）  → 統合検証
Phase 2: コア機能実装（DB・主機能）     → 統合検証
Phase 3: 追加機能（連携・通知・拡張）   → 統合検証 + カバレッジ
```

## 3. 役割分担

| ロール | 向いている作業 |
|--------|----------|
| 監督役（Supervisor） | タスク分割、品質管理、統合検証、教訓の保存 |
| Implementer | コード記述、ファイル作成、機能構築 |
| Reviewer / テストアーキテクト | テスト設計・実装、コードレビュー、検証 |
| Researcher | ドキュメント閲覧、コードベース検索、情報収集 |
| Integrator | 結果統合、競合解消、最終組み立て |

ペルソナは固定ではなく、プロジェクトの規模と性質に応じて調整する。**基本は3体、明確な価値があれば4体**程度が扱いやすい。並列数を増やすほど統合コストと競合リスクが増えるため、体数自体を目的にしない。

**実績のある編成テンプレ:**

- **実装2 + 実機/実データテスト1 + 敵対的レビュアー1** — リーダーが報告を受けてタスクを振り分ける。レビュアーは「ADVERSARIAL third-party reviewer」として承認バイアスを排除する
- **A/B 検証編成** — 修正前後のコードを別ブランチに用意し、**まっさらなエージェント2体**がそれぞれ同一の実データを処理、さらに別のレビュアー（codex 等の別モデルも有効）が時間比較と齟齬チェックを行う
- レビューには「**いつ時点の・どのコミット/成果物まで見たか**」の明記を義務付ける（後から検証根拠を追跡できるようにする）

## 4. エージェント間インターフェースの明確化

各エージェントについて以下を指定する：

- **入力**：エージェントが受け取る情報/文脈
- **出力**：生成すべき成果物（ファイルパス、フォーマット、内容）
- **境界**：所有するファイル/ディレクトリ（競合防止のため）
- **完了シグナル**：成功裏に完了したことを判断する方法

```
Agent A (Implementer):
  Input: Feature spec from task description
  Output: Implementation in src/feature/
  Boundaries: Only modify files in src/feature/
  Done when: All files created, no syntax errors
```

プロンプトは以下を満たすこと：

- **完結性**：必要な文脈をすべて含む。エージェントは途中で質問できない。ユーザー不在で完走する前提（「質問せず最適解で進める」）で、判断に迷う点は妥当なデフォルトを選ばせて前提を記録させる
- **具体性**：期待するファイル・パス・フォーマットを明確に指定する
- **境界明示**：やるべきこと・やってはいけないことを共に伝える
- **検証可能**：出力が正しいかを確認する方法を含む

```
## Task
[Clear one-sentence description]
## Context
[All background information needed]
## Expected Output
[Exact deliverables with file paths and formats]
## Constraints
- Only modify files in [specific directory]
## Verification
- [How to check the output is correct]
```

## 5. 衝突回避ルール

複数エージェントが同一ファイルを同時編集すると競合が発生する。以下のルールで防ぐ：

- **pyproject.toml / package.json** — 依存追加は1エージェントに集約する
- **conftest.py** — テストアーキテクトが所有。他はフィクスチャを使う側に徹する
- **config / 定数ファイル** — 定数追加は1エージェントにまとめる
- **原則** — 各エージェントは自分の担当ファイルのみ作成/編集する

## 6. タスク管理（TaskCreate/Update パターン）

```
TaskCreate → 全タスク作成 → 依存関係設定（addBlockedBy）
→ Phase 開始時: in_progress に更新
→ Phase 完了時: 監督役が検証 → completed に更新
```

タスクの粒度は「1エージェントが1ターンで完了できる」程度にする。大きすぎるタスクは分割し、依存関係で順序を制御する。

## 7. 統合検証（監督役）

各フェーズ完了後、監督役は以下を順に実行する（Python プロジェクトの例）：

```bash
uv sync --extra dev                                        # 1. 依存同期
uv run pytest tests/ -v --tb=short                         # 2. 全テスト
uv run mypy {package}/ --ignore-missing-imports            # 3. 型チェック
uv run pytest tests/ --cov={package} --cov-report=term-missing  # 4. カバレッジ（最終フェーズ）
```

テストや型チェックが失敗した場合、該当タスクの担当エージェントに修正を指示する。**全パスするまで次フェーズに進まない。**

エージェントが失敗した場合：
- 出力を読み、何が間違っていたかを理解する
- プロンプトを修正して再試行するか、手動で処理するかを決める
- 再起動は1度のみ。それ以上は手動処理にエスカレーションする

## 8. 長時間ジョブの監視（停滞検知）

サブエージェントに学習・評価などの長時間ジョブを任せると、**「フェーズ完了 → idle」で止まり、次フェーズに自律的に進まない**停滞が起きる（実例: Stage 1 学習完走後、次の評価を開始せず 20 分停滞）。オーケストレータが受け身でメッセージを待つだけだと気づけない。

**エージェント側への指示（プロンプトに明記する）:**

- 長時間ジョブは必ずバックグラウンド起動し、**起動直後に「PID・ログパス」を即報告**させる
- ジョブ完了ごとに即結果報告。**フェーズ間で手を止めず、次フェーズの起動まで自律的に進む**ことを明示する
- 異常時（発散・成功率0など）の中断・報告条件をあらかじめ与える

**オーケストレータ側の能動監視:**

- エージェントの自己申告に頼らず、**Monitor を常駐**させてログを直接watchする
- 検知は**プロセス名ではなくログファイル基準**にする — `pgrep -f "<ジョブ名>"` は他エージェントが打つ `ps aux | grep <ジョブ名>` 等のコマンド行に**誤マッチ**する。ログの「生成 / サイズ成長 / Traceback / 完了マーカー」を見るのが確実
- 「N分以内にログが生成されなければ警告」を仕込み、警告が出たら (1) 担当へ即時開始を指示、(2) 応答がなければ**専任エージェントを新規投入**して引き継ぐ

```bash
# ログファイルベースの停滞検知ループの骨子（Monitorに渡す）
while true; do
  [ -f "$LOG" ] || { idle=$((idle+1)); [ $idle -eq 12 ] && echo "WARN: not started"; }
  grep -q "DONE_MARKER" "$LOG" 2>/dev/null && { echo "DONE"; break; }
  grep -qE "Traceback" "$LOG" 2>/dev/null && echo "ERROR: $(tail -2 $LOG | tr '\n' ' ')"
  size=$(stat -c %s "$LOG" 2>/dev/null); [ "$size" = "$last" ] && stall=$((stall+1)) || stall=0
  [ $stall -eq 10 ] && echo "STALLED: no growth 10min"; last=$size; sleep 60
done
```

**共有リソース（GPU等）の直列化:**

- GPU重ジョブの**実行所有権は1エージェントに一本化**する。並列に走らせるとロード時OOM等の競合が起きる（unified memory でも CUDA アロケータレベルで衝突する）
- 途中で所有権を移す場合は、旧担当に「新規ジョブ起動禁止＋起動済みジョブの報告」を明示指示してから移譲する（二重実行防止）

## 9. 事後レビュー

タスク完了後、振り返りを行う（efficiency-review スキル参照）：

- 分解はうまく機能したか？エージェント間に不要な依存はなかったか？
- 作業量が偏ったエージェントはなかったか？より少ない体数で同じ結果を得られなかったか？
- 再利用可能なパターンは skill-creator スキルでスキル化またはメモリに保存する
