# issue-log

作業中にぶつかった問題と、その解決方法を**簡潔に**残すためのスキル。
`issue/issue_1.md` という1つのファイルに、問題ごとのエントリを**追記して蓄積**する。
**特定プロジェクトに依存しない** — 固有のコンポーネント名はファイル側（下記「area 一覧」）で定義し、スキル本体は汎用のまま。

## 保存場所

- `issue/issue_1.md` — ここに全エントリを追記していく（`issue/` 直下に置く。下層フォルダは作らない）。
- 200行を超えたら `issue/issue_2.md` を新規作成し、以降はそこに追記する（`issue_3.md` …と続ける）。
- `issue/` は `.gitignore` 対象にしておく（コミットしないローカル知識ベース）。ファイルが無ければ作ってよい。

## ファイル冒頭（プロジェクト固有の area 一覧）

`issue/issue_N.md` の先頭は見出し＋ area 許可リストの2要素:

```markdown
# Issue log (issue_1)

area 一覧: <このリポジトリのコンポーネント名をカンマ区切り>, infra, repo
```

- `area 一覧:` は、このプロジェクトで使ってよい area タグの**許可リスト（統制語彙）**。表記ゆれを防ぐ正本。
- **初回**（一覧が無い/空）に、リポジトリ構成からコンポーネント名を拾って埋める:
  `services/*` `packages/*` `apps/*` `src/<module>/` など**トップレベルの区画名**を、短く一意な名前にする。
- どのプロジェクトでも使える汎用の受け皿として `infra`（ビルド/CI/依存/環境）と `repo`（git運用/命名/docs）を必ず含める。
- 新しい area が必要になったら、**先にこの一覧へ追記してから**エントリで使う（勝手に増やさない）。
- `issue_2.md` 以降に分割したら、冒頭にこの「area 一覧」もコピーする。

## 書式（厳守：簡潔に）

各問題は `##` エントリで追記する:

```markdown
## <問題の短いタイトル> (YYYY-MM-DD)

**問題**: <2行以内で、何が起きたか>

**解決**: <5行以内で、どうしたら直ったか>

area: <冒頭の「area 一覧」から1つ以上（カンマ区切り）>
kind: <下の kind からちょうど1つ>
```

- **問題は2行以内、解決は5行以内**。原因と対処の要点だけ。
- **タイトルに固有名を入れる**（エラーメッセージ・関数名・仕組み名など）。これらは**タグにしない** —
  タイトル/本文を `grep` すれば当たるので、タグを無限に増やさない。
- 日付は今日の日付。新しいエントリは**ファイル末尾に追記**する。

### kind（ちょうど1つ・全プロジェクト共通の固定語彙）

`bug`（不具合） / `perf`（速度・リソース・トークン） / `build`（ビルド・依存・環境構築） /
`config`（設定・env・挙動） / `test`（テスト・検証） / `git`（バージョン管理運用） /
`experiment`（計測・切り分けの知見） / `docs`（ドキュメント・命名規約）

汎用なので基本この8種で足りる。増やさない（足りなければ area 側で補う）。

### area（1つ以上・プロジェクト固有）

**ファイル冒頭の「area 一覧」に載っている名前だけ**を使う。同じコンポーネントは常に同じ綴りで（表記ゆれ禁止）。

## 手順

### 作業開始前（重複チェック — 全文Readしない）

1. まず**見出し + タグだけ**を grep して検索面を作る（本文はコンテキストに載せない）:
   `grep -nE '^## |^area:|^kind:' issue/issue_*.md`
   キーワードが具体的なら直接絞る: `grep -inE '<keyword>' issue/issue_*.md`
   area で絞る: `grep -n '^area:.*<name>' issue/issue_*.md` ／ kind で絞る: `grep -n '^kind: perf' issue/issue_*.md`
2. 候補が出たら、その**行番号周辺だけ** Read する（`offset`/`limit` 指定）。**全文 Read はしない。**
3. 既に同じ問題があれば、新規追記せず**その既存エントリを更新する前提**で進める。

### 作業終了時（記録）

1. 今回の問題と解決を上記の書式でまとめる（area/kind を付ける）。使う area が「area 一覧」に無ければ、先に一覧へ追記。
2. **既存に該当エントリがあれば**: そのエントリを修正する（解決方法が変わった／補足が増えたら更新）。
3. **なければ**: 一番新しいファイルの末尾に `##` エントリを追記する。
4. 追記後、そのファイルが**200行を超えていたら** `issue_(N+1).md` を新規作成し、先頭に見出し＋「area 一覧」を置く。
5. 書式どおり**問題2行・解決5行以内**、kind は固定語彙・area は一覧内のみ。冗長なら削る。

## 原則

- 記録は**簡潔さが命**。コードブロックやコマンドは1〜2行の要点のみ。
- 1エントリ=1問題。複数の問題を1エントリに混ぜない。
- **kind は共通の固定リスト、area は冒頭一覧から**。固有名はタイトルへ（タグを無限に増やさない）。
- 解決していない問題は「解決」に暫定対処と未解決である旨を1行で書く。
- 本文・タイトルは各プロジェクトのドキュメント言語に合わせてよい。
