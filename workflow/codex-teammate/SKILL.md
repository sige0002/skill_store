# Codex Teammate

codex CLI を「別モデルの第2意見」として呼び出し、レビュー・検証・協働作業を行うための定型手順。自分の実装を自分でレビューするより、独立した視点のほうがバイアスなく不備を発見できる。

## 基本レシピ

### レビュー用（読み取り専用・安全）

```bash
codex exec --sandbox read-only --skip-git-repo-check "$(cat prompt.txt)" </dev/null 2>&1 | tee ans.txt
```

### 実作業用（codex に修正までさせる場合）

```bash
codex exec --dangerously-bypass-approvals-and-sandbox "$(cat prompt.txt)" </dev/null 2>&1 | tee ans.txt
```

### 長時間ジョブ（バックグラウンド + ポーリング）

```bash
nohup timeout 420 codex exec --sandbox read-only --skip-git-repo-check \
  "$(cat prompt.txt)" </dev/null > ans.txt 2>&1 &
# 完了検知
pgrep -af "codex exec" || echo "done"
```

## 必須の作法（ハマりどころ）

- **`</dev/null` を必ず付ける** — 付けないと stdin 待ちで固まる
- **`--skip-git-repo-check` を付ける** — git リポジトリ外で実行すると失敗する
- **プロンプトはファイルに書いて `"$(cat prompt.txt)"` で渡す** — インライン長文はエスケープ事故のもと。プロンプトファイルは scratchpad に置く
- **疎通確認を先に行う** — 初回は `codex exec ... "Reply READY"` で生存確認してから本タスクを送る
- **無反応時の方針を決めておく** — タイムアウトしたら1回だけ再送、それでも応答がなければ codex なしで進行する（作業を止めない）

## プロンプト設計

レビュー依頼には以下を含める：

1. **対象の明示** — レビューさせたいファイルパス・ドキュメント・コミット範囲
2. **評価観点** — 「実装不備がないか」「ドキュメントと実装の齟齬」「教材として正確か」など具体的に
3. **証跡の要求** — 「評価はタイムスタンプと、どのコミット/ファイルまで見たかを明記すること」
4. **厳しさの指定** — 甘い承認を避けたいときは「厳しく評価せよ」「敵対的にレビューせよ」と明示する

## 典型ユースケース

- **実装レビュー**: 「docs/ の仕様に従って実行し、エラーや実装不備がないか確認させる」
- **A/B 検証の審判**: 2つの実装/ブランチの出力を比較させ、時間差・齟齬をチェックさせる
- **ドキュメント/教材/スライドの正確性レビュー**: 技術的な誤りの指摘と修正提案
- **設計のセカンドオピニオン**: 「この構成でいいか厳しく評価」— 採用判断の根拠として記録を残す

## 注意点

- codex の回答は独立した意見であり正解ではない。齟齬があれば根拠を突き合わせて自分で判断する
- レビュー結果（ans.txt）は使い捨てにせず、判断根拠として issue-log やコミットメッセージに要点を残す
