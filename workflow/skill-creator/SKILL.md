# Skill Creator

`~/.claude/skills/`（全プロジェクト共通）またはプロジェクトの `.claude/skills/`（プロジェクト固有）配下にスキルを作成・管理するためのガイド。

## スキルとは

スキルは、Claude が特定のタスクを遂行するための再利用可能な指示書。以下のようなパターンをスキル化する：

- 繰り返し使う開発パターン（TDD、DB設計、テスト戦略など）
- 複数エージェント協調の手順
- プロジェクト固有のワークフロー
- ツール連携の定型手順

## ディレクトリ構造

```
skills/
├── {skill-name}/
│   ├── SKILL.md          # 必須: フロントマター + 指示本体
│   ├── references/       # 任意: 補足ドキュメント
│   ├── scripts/          # 任意: 自動化スクリプト
│   ├── templates/        # 任意: テンプレートファイル
│   └── examples/         # 任意: 使用例
```

## 作成手順

### Step 1: 意図の把握

ユーザーの目的を明確にする：

1. **何を自動化/パターン化したいか** — 具体的なタスクやワークフロー
2. **いつ発動すべきか** — トリガー条件（ユーザーの発言やコンテキスト）
3. **出力形式** — 期待されるアウトプット
4. **依存するツール/ライブラリ** — 必要な前提条件

会話の中で繰り返されたパターンがあれば、それを抽出して提案する。

### Step 2: 配置先とスコープの決定

- 複数プロジェクトで使う汎用パターン → `~/.claude/skills/`
- 特定リポジトリでのみ意味を持つ手順 → そのリポジトリの `.claude/skills/`
- プロジェクト固有スキルでも、**固有名（ホスト名・パス・コンポーネント名）をパラメータ化すれば汎用化できないか**を常に検討する

### Step 3: SKILL.md の作成

```markdown
---
name: {skill-name}
description: {スキルの説明。いつ使うべきかを明確に書く。トリガー条件を具体的に。}
---

# {スキル名}

## 概要
{このスキルが解決する問題と、なぜこのアプローチが有効かの説明}

## 手順
{ステップバイステップの指示}

## テンプレート/パターン
{コード例、設定例、定型パターン}

## 注意点
{よくある落とし穴、ベストプラクティス}
```

### Step 4: 品質チェック

- [ ] YAML フロントマターに `name` と `description` が含まれている
- [ ] `description` にトリガー条件が明記されている
- [ ] 指示が命令形で書かれている
- [ ] なぜそうするかの理由が説明されている（MUST/NEVER の乱用を避ける）
- [ ] SKILL.md が 500 行以内に収まっている（超える場合は references/ に分割）
- [ ] コード例が実際に動作する形で記載されている

## 記述ガイドライン

### description の書き方

description はスキルが発動するかどうかを決める最重要フィールド：

- **何をするか** と **いつ使うか** の両方を含める
- 関連するキーワードを網羅的に含める（ユーザーが使いそうな言葉で）
- やや積極的にトリガーするように記述する（発動しすぎより発動しなさすぎのほうが問題）

**良い例:**
```yaml
description: SQLite でデータベース層を TDD で実装するためのパターン集。sqlite3、FTS5全文検索、Upsert、テスト設計が必要なとき、またはPython標準ライブラリのみでDB層を構築したいときに使用する。
```

**悪い例:**
```yaml
description: SQLiteのパターン
```

### 指示本体の書き方

- **理由を説明する** — なぜそのパターンが有効かを伝えると、モデルが応用できる
- **具体例を含める** — 抽象的な説明より、動くコード例が効果的
- **段階的に開示する** — 全情報を一度に出さず、参照ファイルに分割して必要時に読ませる
- 300行以上のリファレンスは `references/`、再利用可能なスクリプトは `scripts/`、テンプレートは `templates/` に分離し、SKILL.md からパスと用途を明示する

## 既存スキルの改修・統合

1. **フロントマターの確認** — name/description が適切か
2. **構造の確認** — SKILL.md + ディレクトリ構造になっているか
3. **内容の確認** — 理由の説明があるか、コード例が最新か
4. **重複の確認** — 類似スキルがあれば統合を検討する。統合時は両者のユニークな知見を落とさない

## サードパーティスキルの取り込み規約

外部リポジトリやマーケットプレイスのスキルを取り込む場合：

1. **取り込む前に `SKILL.md` 全文と `scripts/` を必ずレビューする**（実行可能コードの有無・挙動を確認）
2. 付属の install スクリプトは実行せず、ファイルを手動でコピーする
3. ライセンスを確認し、出典・レビュー日・スキャン結果を `SOURCES.md` に記録する
4. ライセンス不明のものは取り込まない

## スキル化するほどでないパターン → メモリに保存

一度限りに近いが将来参照価値のあるワークフローパターンは、スキルではなく永続メモリ（`~/.claude/projects/<project>/memory/`）に保存する。

保存前に以下を言語化する：

- **What**：何のパターンか（例：「API エンドポイント変更時はハンドラと OpenAPI 仕様の両方を更新する」）
- **When**：いつ適用すべきか（トリガー条件）
- **Why**：なぜ価値があるか
- **How**：ステップバイステップの手順

保存条件：再利用可能な一般性があり、実行可能な具体性があり、少なくとも1回の成功事例で検証されていること。

手順：

1. 既存メモリファイルを Glob/Grep で確認し、類似パターンがあれば新規作成せず既存を更新する
2. `<slug>.md` としてフロントマター（name/category/日付/tags）+ 本文（When to Use / Pattern / Example / Why This Works）で保存する
3. インデックスファイル `MEMORY.md` に1行エントリを追記する
4. 同じパターンがセッションをまたいで3回以上使われたら、スキルへの昇格を検討する
